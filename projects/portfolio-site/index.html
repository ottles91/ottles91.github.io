Introduction
"Do not, under any circumstances, put code that you wrote for any assignment on the internet"

This sentiment—and many others like it—was repeatedly shoved down my throat during my undergrad. I get the intent: they didn't want students copying each other's work. But the messaging was so extreme that, even now, most of the code I wrote in college hasn't been backed up in any meaningful way—not even to a private repository. That's something Im slowly starting to change.

Shortly after graduation, I realized I needed a portfolio to showcase my work—both the projects I created for school and the personal ones I tackled in my free time. While web development has never been a specific career goal, I did take a web programming class in college, so I decided to build a website to highlight what I've been working on.

Goals

LinkedIn is… fine. It's fine. But I really wanted a space where I could share both personal projects and tech-adjacent hobbies—things that don't quite fit with the highly professional (and often slightly stuffy) tone of LinkedIn. I want to present my work while still being unapologetically myself. In building this website, I set out to accomplish a few things that are harder to do on LinkedIn or other career-focused platforms:

  • Create detailed write-ups of the projects I've built
  • Highlight specific skills I've developed
  • Give potential employers and clients a clear picture of who I am and what I can do
  • Give myself an excuse to explore and learn more about web development
  • Have some fun building a website from scratch
Tech Stack

I took exactly one (!) web programming class in college. To be honest, web development never really intrigued me. I felt (and still feel) that markup languages like HTML and XML are just too difficult to read. I also didn't love using three different languages for a single project—it felt a bit unfocused.

Still, I decided to give web programming another fair shot, and honestly, it's turned out okay! I'm still easing back into it, so it was important to me to understand the basics before diving into more advanced web technologies. For now, the tech stack for this site is fairly simple:

  • HTML
  • CSS
  • Vanilla JavaScript
  • Git - Version Control
  • GitHub Pages - Hosting
Design Philosophy

I wanted to create something approachable and minimal, but flexible enough to adapt as my needs evolved. I'm not going to pretend I'm a design expert—many of the decisions I made for this site were borrowed from and remixed based on websites I enjoy visiting. Even the dual-header layout wasn't something I initially planned; it came from paying closer attention to the sites I frequent.

The decision to use a dark background with light text comes from my programming roots: staring at black text on a bright white background all day hurts my soul. I wanted to emulate the color scheme of my code editor, but with a bit more visual polish. Some of my less tech-savvy friends don't quite get the appeal, so I may add a theme toggle at some point to appease them—even if I think they're wrong 🙃

I also wanted the design to work well on mobile devices. Responsive design was a priority, so I've made sure the site scales elegantly—even on the smallest phones currently available.

Key Features

While building this site, I ended up creating a number of small—but meaningful—features that help everything feel a bit more polished. Many of them are easy to overlook when they're working well, but once they're gone, you definitely notice. Below are some of the key features I've added to improve usability, accessibility, and the overall user experience.

Responsive Design

One of the first major design updates I tackled was ensuring the website performed well on phones and tablets. For the most part, this meant writing specific CSS rules for cases where the screen size dips below a certain threshold. These adjustments help keep text readable on small screens and ensure that key elements—like images—render in a way that improves visibility and usability.

I also created desktop-specific CSS rules for images. On certain pages, text is allowed to "wrap" around the right side of an image when viewed on a desktop. On mobile devices, those same images and text elements are stacked vertically instead. This provides a more dynamic layout on larger screens, while prioritizing readability and flow on smaller ones. This allows my website to take advantage of the strengths that are unique to each form factor.

On the desktop version of the site, there's enough horizontal space to display links to different site sections directly in the top navigation bar. On mobile, however, screen space is more limited. To solve this, I implemented a “hamburger menu” that keeps navigation accessible without taking up unnecessary real estate.

This approach lets me store the header and footer in separate HTML files and insert them dynamically at load time. It's a simple and effective way to follow the DRY (Don't Repeat Yourself) principle—without relying on any frameworks or build tools.

Code Snippets

I wanted a way to share code snippets that preserved the formatting and readability you'd expect from a modern IDE. While HTML includes basic <code> tags, they didn't offer the level of clarity or control I was looking for.

I turned to Highlight.js to format and highlight my code blocks. Highlight.js provides syntax-aware highlighting across multiple languages, making my code easier to read and more visually engaging for anyone familiar with programming.

CSS Theming

To keep things visually cohesive and easy to tweak, I rely heavily on CSS custom properties (variables). These are defined at the root level and serve as a central source of truth for values like colors, fonts, and spacing. This allows me to apply a consistent theme across the site while making future changes fast and painless—if I want to try a new accent color or adjust padding, I only need to update a single variable.

For example, my color palette includes variables like --header-bg, --footer-bg, and --link-color-body, which control styling for site-wide elements. I also use variables for typography and layout—such as --font-family-header and --padding—to keep everything flexible and readable.

This approach not only improves maintainability, but also lays the groundwork for future enhancements like light/dark mode toggles or user-selectable themes, which could be implemented simply by swapping variable values in a different theme context.

Design & UX Decisions

While I'm building this website primarily to showcase my technical skills, it was also important to me that the design feel approachable and familiar to all users. I wanted the basic building blocks—especially the header and footer—to reflect patterns visitors already recognize from other websites. My initial header was very simple: a single horizontal row of navigation links.

Shortly after setting that up, I started paying closer attention to how other sites approached their header design. I noticed a common trend—many sites use a stacked (or dual-layer) header, with the top layer focused on branding and the bottom layer dedicated to navigation. I decided to adopt that same approach for my site. Not only did it help reinforce my personal branding, but it also aligned my layout more closely with the design patterns users already expect to see online.

Conclusion & Future Updates

While I initially envisioned this website as a simple way to showcase my personal projects, it's grown into something much more meaningful. I've found that I genuinely enjoy working on it—it's become a space for creativity, learning, and continual refinement. I'm always coming up with new tweaks and ideas to push my web development skills further and improve the site overall. My hope is that this site will keep evolving to meet my needs for years to come.
