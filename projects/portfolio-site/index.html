Responsive Layout

The site is built to look great across a wide range of devices, from large desktop monitors to compact mobile phones. I've used flexible containers and relative units to allow content to resize smoothly, along with targeted media queries to adjust layouts at common breakpoints (e.g., 768px for tablets, 480px for phones).

On smaller screens, navigation switches to a hamburger menu for better usability, and certain elements like social icons shift position to avoid clutter. Font sizes, padding, and margins adapt to improve readability and touch accessibility. While the layout is responsive, I've also tried to ensure that performance and visual hierarchy are preserved regardless of the screen size.

Dynamic Header and Footer Loading

Rather than repeating HTML across every page, I use JavaScript to dynamically load shared header and footer content. Every page on my website includes the following HTML snippet:

<div id="header-placeholder"></div>

  1. Takes a CSS selector (such as #header-placeholder)
  2. Fetches an external HTML file (like header.html)
  3. Injects the content of that file into the matching element on the page
  4. Optionally executes a callback function after the injection

This allows me to store the header and footer in their own separate HTML files and dynamically insert them into each page at load time. It's a clean way to follow the DRY (Don't Repeat Yourself) principle while sticking to plain HTML, CSS, and JavaScript.

CSS Theming

To keep things both visually cohesive and easy to tweak, I rely heavily on CSS custom properties (variables). These variables are defined at the root level and act as a central source of truth for common values like colors, fonts, and spacing. This allows me to apply a consistent theme across the site while making future changes fast and painlessâ€”if I decide to try a different accent color or adjust padding, I only need to update a single variable.

For example, my color palette includes variables like --header-bg, --footer-bg, and --link-color-body, which control site-wide elements. I also use variables for typography and layoutâ€”such as --font-family-header and --paddingâ€”to keep things flexible and readable.

This approach not only promotes maintainability, but also lays the groundwork for future enhancements like light/dark mode toggles or user-selectable themes, which can be implemented simply by swapping variable values in a different theme context.

Challenges

This has been both my biggest overall web project as well as the first web project I've taken on in several years. As of the time of writing I've been working on this project for a little over a month now, with multiple commits to my repository during each weekday. Even at this early stage I've overcome multiple challenges and learned many new skills that have helped me learn and grow as a developer.

Responsive Design

One of the first large updates I tackled related to design was to ensure that the website performed well on phones and tablets. For the most part this simply involves creating specific CSS rules for instances where the screen size dips below a certain threshold. This helps ensure that text remains readable on small screens, and specific elements, like images render in a way that increases their visibility on smaller screens

I created some desktop-specific CSS rules around images. On certain pages, the page content will be allowed to "wrap" around the right side of an image on the desktop version of the page, but on mobile the images and text will be stacked vertically. This allows for a more dynamic layout that better takes advantage of the additional screen space available on desktop browsers, while also allowing the mobile site to prioritize readability for a small screen.

On the desktop version of the website I have enough room to display links to the different site sections across the top of the navigation header. On mobile, horizontal space is much more limited. Because of this I implemented a "Hamburger Menu" on mobile, which keeps the different sections of the site accessible without taking away from screen space unnecessarily

Dynamic Content Injection

Because I'm relying on vanilla HTML, CSS and JavaScript (at least for this inital version of the site) I wanted a way to reduce the amount of repeated code segments on the site. I quickly realized that both my site header and footer were always going to be universal elements that wouldn't change between pages. I didn't want to have update them across every page whenever I wanted to make a small change to their implementation. To address this, I created a system that dynamically injects my stand-along header and footer HTML files into each page.

Code Block Formatting

As discussed previously, I wanted a system that allowed me to share snippets of my own code in a way that maintained the formatting and readibility that you'd find in any modern IDE. HTML does include support for simple code tags, but that didn't give me the depth and flexibility that I was looking for. As dicussed previously, I turned to Highlight.js to help me format and highlight my code blocks in a way that would be readable to anyone familiar with programming.

Design & UX Decisions

While I am creating this website largely to showcase my technical skills, it was important to me to ensure that I created a design that was approachable to all users. I wanted the basic building blocks, especially the header and footer to be familiar to what visitors might see on other websites. My initial header was very simple, simply containing a list of links to different sections of the site arranged hoizontally across the header itself.

Shortly after creating my initial header I started paying attention to how header designs on other websites differed from my own. I noticed that a popular design norm was to have two layers to the header: The top layer focusing on branding, and the lower focused on navigation links. I decided that I wanted to implement this stacked (or duel-layer) header design on my own site to help establish my own personal brand, as well as to try to stay within in the norms of general website design that I observed elsewhere on the internet.
